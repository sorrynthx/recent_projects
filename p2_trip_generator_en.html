Four processing stages
STAGE 1 — Ingestion
Log Caching
We scan all raw GPS log files and compress them into a per-route cache. If the cache already exists for that date, we skip the scan entirely.
Why: re-scanning hundreds of MB on every run was too slow.
STAGE 2 — Loading
Data Merge
The cache and DTG database are merged into one clean dataset. Duplicates are removed, and coordinates stored as integers are divided by one million to get real GPS values.
Why: two sources overlap — without deduplication, the same event gets counted twice.
STAGE 3 — Analysis
Trip Analysis (parallel)
Four processes run at the same time — one per route. Each is fully isolated, so if one route fails, the others keep running.
Why: Python's GIL blocks true parallel threads — separate processes bypass that.
STAGE 4 — Export
Reporting + DB Load
Results are written to CSV first. DB import is a separate step — so you can check the output before touching the database. Failed imports can be retried without re-running the analysis.
Why: separating the two steps makes debugging much easier.
Key design decisions
Honest Track principle
If there is no GPS evidence for a stop, we record None — not an estimate, not an interpolation. The output has to be legally defensible.
Analysis vs. DB export separation
Run analysis first, verify the CSV output, then trigger DB import with a separate flag. Failed imports can be retried without redoing the full analysis.
Service type environment variable
One variable controls which database and Redis endpoint is used — city, village, or test. This prevents writing data to the wrong service by accident.
Redis single scan
Old: one Redis query per route, every time. New: one scan at startup builds all the maps we need and automatically detects new routes.
Stage 1 — Trip separation (splitter)
How does the splitter know when a new trip starts?
The idea is simple: when a bus returns to the origin area, a new trip begins. We set a 100-meter radius around the origin. When the GPS position crosses back inside that circle, we mark a trip boundary.

Three force-split rules
10-minute cooldown — If the bus re-enters the origin within 10 minutes of leaving, we ignore it. The route might pass near the depot mid-trip. Without this guard, a short loop near the origin would create a false split.
30-minute GPS gap → force split — If GPS goes silent for over 30 minutes, the device was probably powered off. We cut the trip there. Continuing the same trip across a 30-min gap would produce wrong arrival times.
20-minute dwell at origin → force split — If the bus stays inside the origin area for over 20 minutes, it is waiting for the next departure. Long dwell at the depot means one trip ended and another is about to start.
Minimum travel distance — If the bus never travels far enough from the origin, we discard that candidate. It is depot noise, not a real trip. GPS drift at the depot can look like movement — this rule filters it out.
Stage 2 — Stop matching (timeline)
How do we find when the bus passed each stop?
①
Bounding box filter — Draw a box around the stop. Only GPS points inside the box move forward. This avoids running heavy calculations on every point.
②
Precise distance check — For the filtered candidates only, calculate the exact distance to the stop center. Points within the radius become hits.
③
Clustering — GPS hits within 30 seconds of each other are grouped into one cluster. One cluster = one visit to that stop.
④
DP optimization — Each cluster gets a score. DP finds the highest-scoring sequence that respects stop order and time order. Details below.
⑤
2-pass check — After DP, we scan forward and backward once more to catch any timestamps that are still out of order.
① Raw clusters — After hit scanning and clustering, each stop may have multiple cluster candidates. Stop C has none — GPS never entered its radius.
DP score function
Each cluster gets a score based on three things. DP finds the path with the highest total score — with two hard rules: stop order must go forward, and time must go forward.
dist_score = max(0, 100 − distance × 1.5) ← closer to stop = higher score
size_score = min(100, point_count × 2) ← more GPS hits = higher score
bonus = +20 if source is RAW data ← real measured data gets a bonus
─────────────────────────────────────────
Two rules: stop order must go forward AND time must go forward
If a stop has no cluster — no GPS data within the radius — it is recorded as None. No guessing. No interpolation.
Parallel execution model
CPU BOUND — Route Analysis
Process Pool
Python has a lock called the GIL — it prevents true parallel execution in threads. To get around that, we use separate processes. Each route runs in its own process with its own Python interpreter.

Bounding box filter — how it works
Before calculating the exact distance to a stop, we draw a simple box around it. Points clearly outside the box are skipped — no math needed. Only points inside go through the full distance check.
① BBox filter — Grey points are outside the bounding box and skipped immediately. Only blue points inside the box move to the next step.
Cache strategy
💾
Cache skip
If a cache file already exists for that date, we skip the entire log scan. A force-rebuild flag is available when needed.
🌙
Midnight window
We load three days of logs — yesterday, today, and tomorrow — to capture late-night runs that cross midnight.

Key phrases for the interview
"Python has a lock called the GIL — it blocks true parallel execution in threads. To get around that, we used a process pool. Each route runs in its own process, so a failure in one never affects the others."
"The DP algorithm finds the best stop sequence by enforcing two rules: stop order must go forward, and time must go forward. The bus cannot visit stop 10 before stop 5, and timestamps cannot go backward."
"If there is no GPS evidence for a stop, we record None — no estimate, no interpolation. The data has to be legally defensible."
"We originally capped DB export at two threads to avoid lock collisions. After switching to bulk inserts — one thousand rows per query — the lock pressure disappeared. So we removed the limit entirely."
"Before, we ran one Redis query per route every time the pipeline started. Now, one scan at startup builds all the maps we need and auto-detects new routes."
