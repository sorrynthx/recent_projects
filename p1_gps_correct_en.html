Key design decisions
Manual polling
We wrote a single-thread loop ourselves instead of using Spring's built-in listener. This avoids unnecessary context-switching at high load.
Why: 1,600 vehicles per second — every extra overhead adds up.
Fixed consumer name
Each consumer is named with hostname and index. This prevents ghost consumers from appearing in Redis after a restart.
Why: random names on restart create zombie entries that never get cleaned up.
Read from last on restart
When the service restarts, we skip any backlog and jump straight to the latest records. Real-time visibility is always the priority.
Why: showing 3-minute-old data after a restart is worse than a brief gap.
Distributed lock
A daily normalization job runs at 4:15 AM. A Redis-based lock ensures only one instance executes it, even when multiple pods are running.
Why: without a lock, every pod would run the job at the same time.

Mode selection · Modusauswahl
Route ID in packet?
YES → Mode B (Map-Matching) NO / snap failed → Mode A (EMA Fallback)
Route ID present → Mode B  |  No route ID or snap failed → Mode A (fallback)
MODE A — EMA Smoothing · Glättung (Fallback)
Speed > 140 km/h → discard the recordClearly a sensor error — no bus travels that fast.
GPS jump > 300 m in one second → discardA position jump that large is impossible at normal speed.
Heading change > 120° at speed → discardA bus cannot turn that sharply while moving.
Alpha = 0.5 when moving, 0.05 when stoppedWhen moving, trust the current reading. When stopped, let the historical average dominate — so GPS drift doesn't show fake movement.
MODE B — Map-Matching · Kartenabgleich
Zero-Motion Bypass: speed ≤ 3 km/h + drift ≤ 2 m → skip snap entirelyA stopped bus doesn't need map-matching. Reuse previous coordinates at zero CPU cost — and eliminates ghost speed.
Fast distance check — no square root — 10× faster node searchFor finding the nearest route node, we only need to compare distances — not calculate them exactly. Dropping the square root keeps the order the same.
Heading difference > 80° → fall back to Mode AIf the bus direction and the route direction don't match, the snap would land on the wrong segment.
Distance > 100 m from route → abandon snapToo far off the route means the GPS is unreliable. Mode A handles it instead.
Raw GPS (noisy) EMA smoothed path Road (no route data in Mode A)
Moving · alpha = 0.50 — High alpha means we trust the current GPS reading more. The smoothed line tracks closely but removes sudden spikes.
Raw GPS Snapped to route Bus route Mode A fallback point
Normal Snap — Raw GPS is off the route. The engine finds the nearest segment and snaps the coordinate onto it. All checks pass → snap succeeds.

Key optimizations
⚡
Redis Pipeline Batching · Pipeline-Bündelung
BEFORE
200 SET = 200 round-trips
→
AFTER
1 pipeline call = 1 round-trip
All 200 SET commands are bundled into a single network write. Redis processes them in order and replies once. Result: 200× fewer round-trips.
Click a button above to see the difference in network I/O.
♻️
Zero-Allocation · GC-Druckreduktion
BEFORE
new VehState() per record
→
AFTER
mutate in-place, reuse object
We pre-allocate the state object at startup and just update its fields. We also replaced the time API call with a lightweight millisecond counter — no heap allocation at all. GC pauses dropped to near zero.
📐
Fast Distance Check · Schnelle Näherungsberechnung
BEFORE
Haversine (sin / cos / sqrt)
→
AFTER
fastDistanceSq (arithmetic only)
For finding the nearest route node, we only need to compare distances — not calculate exact values. Dropping the square root keeps the same ordering but runs about 10× faster.
⏸️
Zero-Motion Bypass · Stillstandserkennung
BEFORE
Full snap computation while stopped
→
AFTER
Skip if spd ≤ 3 km/h + drift ≤ 2m
When a bus is stopped, we skip the entire map-matching step and reuse the previous result. Zero CPU cost — and ghost speed is completely eliminated.

Thread pool
core-size12
max-size16
queue-capacity100
Intentionally below the 20-core CPU count — this keeps context-switching low. A queue of 100 absorbs traffic spikes without dropping anything.
Redis connection pool
max-active40
max-idle30
share-nativefalse
40 parallel channels — no thread ever has to wait for a Redis connection. We also disabled connection sharing, which removes a bottleneck that shows up in multi-thread environments.
Why 40? We tested 10, then 20. At 40, no bottleneck — enough headroom.
